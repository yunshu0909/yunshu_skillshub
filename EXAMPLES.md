## 🎨 配图助手 (Image Assistant)

### 场景：为一篇技术文章配图

```
用户: 我写了一篇介绍 AI 工具选型的文章，需要配几张图
助手: [自动触发配图助手流程]
助手: 好的！让我先了解一下需求：
     1. 这篇文章主要用在哪里？（PPT、公众号、博客等）
     2. 谁来看？（技术人员、产品经理、普通用户？）
     3. 你希望图上是更简洁少字，还是信息密度高一些？

用户: 用在公众号文章里，面向产品经理和技术人员，希望简洁易读
助手: 明白了！让我看看文章内容...

     【第2步：配图规划】
     根据你的内容，我建议做 3 张图：
     1. 封面图：AI 工具选型的整体框架
     2. 对比图：开源 vs 商业方案的对比
     3. 决策流程图：如何选择合适的工具

     这样可以吗？

用户: 可以！
助手: 【第3步：文案定稿】
     第1张-封面图的文案：
     - 大标题：AI 工具选型指南
     - 副标题：场景→需求→选型
     - 3个关键词：性能、成本、生态

     确认这些文案吗？

[... 继续逐张确认文案 ...]

助手: 【第4步：提示词输出】
     第1张图的生图提示词：

\`\`\`
16:9横版信息图。奶油纸底+彩铅水彩手绘风格。
中心大标题【AI工具选型指南】，下方流程：场景分析→需求定义→工具选型。
底部3个图标+关键词：性能⚡、成本💰、生态🌱。
少字高可读，干净趣味。
\`\`\`

     可以直接复制提示词去生成图片了！需要调整吗？
```

**预期输出**:
- 图片清单（几张图/每张目的）
- 每张图的文案定稿（Copy Spec）
- 可复制的生图提示词
- （可选）批量生成的图片

```
用户: 我觉得很多产品设计太复杂了，用户根本不知道怎么用
助手: 很好的观点！继续说说，你有遇到过什么具体的例子吗？

```

**预期输出**:
- 洞察记录文件（insights.md）
- 写作记录文件（writing-record.md）
- 最终成稿文章

## 🔄 需求变更工作流 (Requirement Change Workflow)

### 场景：修改 Chrome 扩展的侧边栏交互

```
用户: 我想改一下侧边栏的交互，现在点击按钮后直接跳转，我想加个确认弹窗
助手: [自动触发需求变更流程]
助手: 好的，让我们先明确一下这个变更的范围。

     【第1步：澄清变更】
     - 目标：在侧边栏按钮点击后，增加确认弹窗
     - 不改变的：跳转的目标页面、现有数据流
     - 验收标准：
       1. 点击按钮后显示确认弹窗
       2. 点击"确认"后执行原有跳转
       3. 点击"取消"后关闭弹窗

     这样理解对吗？

[... 继续执行基线确认、影响评估、设计提案等步骤 ...]
```

**预期输出**:
- 需求变更简报（change brief）
- 影响评估报告
- 实施后的代码变更
- 回归测试结果
- 决策日志

## 🔍 GitHub 开源项目搜索助手 (GitHub Repo Search)

### 场景：寻找 Agent 记忆相关的开源项目

```
用户: 帮我找找 agent 记忆相关的开源项目
助手: [自动触发 GitHub 项目搜索流程]
助手: 好的！让我先确认一下需求：

     【第1步：需求收敛】
     1. 主题：agent 记忆（长期记忆、上下文管理、记忆存储等）
     2. 数量：Top 10
     3. 最低 stars：>= 100
     4. 排序模式：相关性优先 / 星标优先？
     5. 目标形态：
        - 可直接使用的产品？
        - 可二次开发的框架？
        - 资料清单/方法论？

用户: Top 10，相关性优先，主要是想找可二次开发的框架
助手: 明白了！需求确认如下：

     核心诉求：
     - 主题：agent 记忆
     - 数量：Top 10
     - 最低 stars：>= 100
     - 排序模式：相关性优先
     - 目标形态：可二次开发的框架
     - 偏好：无
     - 排除：无

     确认吗？

用户: 确认
助手: 【第2步：检索词拆解】
     我将使用以下检索策略：

     Query-1: "agent memory framework"
     目的：高召回核心主题

     Query-2: "llm long-term memory"
     目的：补同义词盲区

     Query-3: "agent memory python sdk"
     目的：聚焦可开发框架

     [继续执行检索...]

助手: 【第3步：检索执行】
     检索完成：
     - 原始召回：47 个仓库
     - 去重后：38 个
     - 硬过滤后：23 个

助手: 【第4步：质量精炼与分类】
     经过噪音剔除和综合排序，选出 Top 10：

     | 仓库 | 星标 | 仓库归属类型 | 项目介绍 |
     |---|---:|---|---|
     | mem0ai/mem0 | 24k | 记忆层/上下文基础设施 | 为AI助手和agent提供长期记忆能力... |
     | ... | ... | ... | ... |

     完整报告已生成，包含：
     - 需求摘要
     - 检索词清单
     - 筛选规则
     - Top 10 详细推荐
     - 下一步建议
```

**预期输出**:
- 需求摘要
- 检索词清单（5-10 组 + 目的）
- 筛选与重排规则
- 结果总览（原始召回/去重后/过滤后）
- Top N 单榜表格（含仓库类型、项目介绍、链接）
- 结论与下一步建议

## 💡 提示

### 自动触发
大多数情况下，你只需要用自然语言描述你的需求，相关的 Skill 会自动触发：

```
❌ 不需要这样：先输入 /thought-mining，然后说"我想写文章"
✅ 直接这样说：我想写一篇关于 XX 的文章
```

### 手动触发
如果自动触发不生效，可以使用斜杠命令：

```
/image-assistant     # 启动配图助手
/thought-mining      # 启动思维挖掘
/prd-doc-writer      # 启动 PRD 撰写
/req-change-workflow # 启动需求变更
/github-search       # 启动 GitHub 项目搜索
```

## 📦 需求池管理 (Backlog Manager)

### 场景 1：记录一个新想法

```
用户: 我想做个功能，让用户可以导出周报数据
助手: [自动触发需求池管理流程]
助手: 好的！让我先了解一下这个想法：
      1. 现在遇到什么问题了？什么场景下需要导出周报数据？
      2. 多久会碰到这个需求一次？
      3. 不做这个功能的话，你现在怎么解决？

用户: 现在周报数据只能在页面上看，领导想让我汇总月度数据，我只能手动复制粘贴
助手: 明白了！痛点是手动复制粘贴效率低，容易出错。
      我理解的是：需要一个导出功能，把周报数据导出为 Excel/CSV 方便汇总分析。
      对吗？

用户: 对的
助手: 好的，我将这个需求归档到"可以做了"区域。需要时随时可以启动开发。
```

**预期输出**:
- `docs/需求池.md` 新增条目
- 总览表更新
- 详情区记录痛点、方案、备注

### 场景 2：整理需求池

```
用户: 整理一下需求池
助手: 好的！让我看看当前的需求池状态...

     【逐条检查】

     #1 - 导出周报数据 (可以做了)
     建议：保持不变，痛点清晰，随时可以启动

     #2 - 暗黑模式 (先放着)
     建议：升档到"需要想想"。已收到 3 个用户反馈想要这个功能

     #3 - 接入 AI 自动写周报 (需要想想)
     建议：保持不变，还需要调研 API 成本和可行性

     #5 - 邮件通知功能 (可以做了)
     建议：降级到"先放着"。半年内没有用户提过这个需求

用户: #2 同意升档，#5 同意降级
助手: [更新需求池文件]
```

### 场景 3：挑选下一个版本做什么

```
用户: 下一个版本做什么
助手: 好的！让我看看"可以做了"的需求池...

     当前有 2 个候选：

     #1 - 导出周报数据
     #4 - 支持多团队切换

     让我用三个标准分析一下：

     | 标准 | 导出周报 | 多团队切换 |
     |------|---------|-----------|
     | 频率 | 每月 1-2 次 | 每天多次 |
     | 可绕过性 | 手动复制，很繁琐 | 必须登出重登，很麻烦 |
     | ROI | 中 | 高 |

     建议：优先做多团队切换，痛点频率高且无法绕过。
     但最终由你决定。

用户: 同意，先做 #4
助手: 好的！已将 #4 标记为"进行中"。需要我启动设计探索流程吗？
```

## 🎨 新功能设计探索 (Design Exploration)

### 场景：设计一个 MCP 工具管理界面

```
用户: 我想做个 MCP 工具管理界面
助手: [自动触发设计探索流程]
助手: 好的！让我们先明确一下需求边界：
      1. 现在遇到了什么问题？为什么想做这个？
      2. 最典型的使用场景是什么？
      3. 这个版本要做什么？明确不做什么？

用户: 现在 MCP 工具配置散落在不同文件里，想集中管理
      主要是查看、启用/禁用、配置参数
      不做编辑配置文件的功能，先只读
助手: 明白了！需求收敛如下：

      做什么：
      - 集中展示所有 MCP 工具
      - 支持启用/禁用工具
      - 查看工具配置参数
      - 区分 stdio/http 类型

      不做什么：
      - 编辑配置文件
      - 添加/删除工具
      - 测试连接

      载体：Claude Code 内部页面

      确认吗？

用户: 确认
助手: 【第3步：ASCII 批量探索】
      我出 5 个不同思路的方案：

      方案 A：卡片列表 + 开关
      ┌──────────────────────────────────┐
      │ ┌────────────────────────────┐   │
      │ │ pencil       [✓] CC [✓] CX│   │
      │ └────────────────────────────┘   │
      └──────────────────────────────────┘

      方案 B：表格视图，一行一个
      ┌──────────────────────────────────┐
      │ 名称       类型   CC    CX      │
      │ pencil     stdio  [✓]   [✓]    │
      │ github     http   [✓]   [✓]    │
      └──────────────────────────────────┘

      [方案 C、D、E...]

      你喜欢哪个方向？

用户: B 的结构不错，但开关用 A 的样式
助手: 【第5步：HTML 设计稿】
      [生成 HTML mockup，使用真实数据]

      效果满意吗？需要微调吗？

用户: 间距稍微调大一点
助手: [微调间距]

     【第6步：全状态覆盖】
     生成完整的设计参考文档，包含：
     - 正常态、加载态、空态、错误态
     - Toast 反馈汇总
     - 边界情况（长名称、大量数据）
     - 交互行为规则表

     【第7步：归档】
     三个文件已输出到 设计/v0.11-MCP管理/
     - 需求总结.md
     - mcp-manager-设计稿.html
     - mcp-manager-全状态设计参考.html
```

**预期输出**:
- `设计/v{版本}-{模块名}/需求总结.md`
- `设计/v{版本}-{模块名}/{模块名}-设计稿.html`
- `设计/v{版本}-{模块名}/{模块名}-全状态设计参考.html`

## 🧠 记忆系统初始化 (Memory Init)

### 场景 1：交互式初始化

```
用户: 初始化记忆
助手: [自动触发记忆系统初始化流程]
助手: 好的！让我先检查一下当前目录...

     当前目录下未发现 CLAUDE.md、MEMORY.md 或 memory/，可以直接初始化。

     让我了解一下你的情况：
     1. 你的角色是什么？（如：SA/售前、开发、产品、运营）

用户: 产品经理
助手: 2. 你主要用 Claude 做什么？

用户: 写 PRD、做竞品分析、整理需求
助手: 3. 对话风格偏好？（简洁直接 / 详细展开 / 随意聊天）

用户: 简洁直接
助手: 4. 有什么特别的工作习惯要我知道的？（可选，可以跳过）

用户: 跳过
助手: 好的！正在生成...

     已生成：
     - CLAUDE.md（人设 + 记忆协议）
     - MEMORY.md（长期记忆）
     - memory/（每日记忆目录）

     使用方式：
     - 正常对话即可，我会自动读取记忆
     - 对话结束时我会问你要不要保存要点
     - 说"记住这个"可以随时存记忆
     - 积累一段时间后说"整理记忆"做一次蒸馏
```

### 场景 2：Quick 模式快速初始化

```
用户: /memory-init quick
助手: [自动触发快速初始化]
助手: 检查当前目录... 未发现已有文件，开始快速初始化。

     已生成：
     - CLAUDE.md（使用默认人设模板）
     - MEMORY.md（长期记忆初始模板）
     - memory/（每日记忆目录）

     默认人设：你是用户的工作搭档，不是通用助手。
     你可以随时编辑 CLAUDE.md 来自定义人设和风格。
```

**预期输出**:
- `CLAUDE.md`（人设 + 对话风格 + 思考方式 + 记忆协议）
- `MEMORY.md`（长期记忆初始模板）
- `memory/` 目录（每日记忆存放目录）
