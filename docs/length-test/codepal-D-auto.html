AI 编程工具（Claude Code / Codex / Cursor / Trae）越用越多，但没有一个工具会管别的工具的事。CodePal 就是那个"连接层"：让 Skills 在多工具间同步、多账户秒切、用量一眼可见、历史对话翻得回来。一句话——AI 写代码，CodePal 写代码之外的一切。

第一幕 · 摩擦

最先察觉问题的人，往往是用得最多的那个。

想象一个每天和 4 个 AI 编程工具打交道的开发者。Claude Code 写主仓代码，Codex 跑实验脚本，Cursor 处理前端微调，Trae 偶尔被拉来对照参考。每个工具都很聪明，每个工具也都很"自我"——它们各自管各自的世界。

于是这个开发者每天都在做一些"AI 不会替他做"的事：

· 同一个 Skill 写了一份，要复制到 Claude、Codex、Cursor 三个目录，改一处全得跟着改。
· Codex Plus 账户额度按 5 小时 + 7 天双窗口算，一触顶要登出 → 换邮箱 → 重登 → 二次验证，1 到 2 分钟，一天来回好几次。
· 月底想知道 Token 花了多少钱、哪个模型最烧，得翻日志手算。
· 上周和 AI 聊过的一段关键对话，今天想接着聊，找不到 Session 文件在哪。
· 新建项目想要标准的 CLAUDE.md 和 .gitignore，每次手动从老项目里抄。

这些事没有一件是"AI 不能做"——而是"AI 工具的边界刚好在它写完那行代码的位置"。再往外一步，就是用户自己的责任。

所以问题不是 AI 不够强，是 AI 之间的缝隙没人补。

第二幕 · 主角登场

CodePal 不写代码，它管"写代码这件事"。

CodePal 是一个 macOS 桌面应用（Electron 套 React，Apple Silicon 优先），定位是 AI 编程工具的"幕后助手"。它不替你思考，不替你生成代码，也不试图取代任何一个 AI 工具。它做的是大家都不愿意做、但每天都要做的那些事——把多个 AI 工具拧成一束。

这个比喻可能更清晰：如果 Claude Code、Codex、Cursor 是几位各自独立的合作者，CodePal 就是那个帮他们对账、排班、同步白板、记备忘的"项目助理"。它不上场，但少了它，几位主角的协作成本会高很多。

容易误解的地方：CodePal 不是又一个 AI 编程工具，也不是 AI 编程工具的"插件"。它是工具之间的层，是用户视角下的"AI 协作面板"。

第三幕 · 主角的四件武器

所有功能按使用场景归在 4 个模块里，按需拿出来用。

这是 PM 视角下最值得记住的一张表——4 个模块 × 各自痛点 × 各自能力。看完它，你大致就握住了 CodePal 的产品骨架。

模块 用户痛点 核心能力
账户与用量 多账户切换费时、额度不知道剩多少、月底不知道花了多少 Codex 多账户秒切（<200ms）+ 凭证后台保活；Claude 额度可挂状态栏；Token 用量按日/周/月聚合 + 费用估算
文档 历史对话翻不到、Markdown 文档散在多个目录 对话回顾（按项目分组 + 全文搜索 + 一键 resume）；多目录 Markdown 浏览（树形 + 内置渲染）
技能中心 Skills 在多工具间难同步、MCP 配置要手改 JSON Skills 中央仓库 + 一键推送所有工具 + 5 分钟自动增量同步；MCP 启用/停用 GUI 化
工具设置 Claude Code 启动权限要每次设、新项目要从零搭、网络抽风排查麻烦 4 种启动模式一键切换；项目模板一键生成；IP 监控 + API 连通性检测

4 个模块之间是并列关系——你可以只用其中一个（比如只用账户切换），也可以全开。它不强制工作流，更像一个工具箱：你今天想用扳手就用扳手，不用为此学习一整套方法论。

反例提醒：如果你试图把 CodePal 当成"AI 编程的全能工作台"——错了。它没有编辑器、没有终端、不跑模型。它的价值恰恰在于不替你做主业，只补你做主业时的缝。

第四幕 · 主角的脾气

这是一个"用 → 痛 → 解决 → 沉淀"飞轮长出来的产品。

读 CodePal 的版本历史，会发现一个明显的节奏：每个版本都对应一个非常具体的痛点。

v1.2.4 网络诊断 —— 因为某次 VPN 抽风，作者排查了半天。
v1.2.6 对话回顾 —— 因为想找一段历史对话翻不到。
v1.4.5 启动历史对话 —— 因为找到对话之后还得手敲 cd 和 --resume。
v1.5.0 Codex 多账户切换 —— 因为切账户太烦了。
v1.6.2 凭证保活 —— 因为切回去发现 token 过期被踢。

作者明确写在 README 里：不提前规划功能列表。这意味着 CodePal 的产品哲学是"先有痛，再有功能"，而不是"先画路线图，再倒推痛点"。从 PM 视角看，这有两层含义：

第一，每个模块都非常贴用户——因为作者本人就是最重度用户，新功能在自己身上跑过几十次才会沉淀进去。第二，它不太会有"PM 提需求"那种发散——想让它做什么，最有效的方式是讲一个真实的痛点故事，而不是列一张能力清单。

顺带一提隐私态度：v1.6.2 加了"凭证保活"——会在本机调 OpenAI 官方的 OAuth refresh endpoint 续签 token。README 专门写了一段隐私声明：只用本地 refresh_token、不上传任何凭证、不代理模型推理、User-Agent 显式标识为 CodePal。这个细节挺能说明产品调性：愿意花一段文字解释一个可能让用户不安的技术行为，而不是默默做掉。

读完你手里多了什么

如果有人问你 CodePal 是什么，你现在可以这样回答：

"它是 AI 编程工具的连接层。AI 工具负责写代码，CodePal 负责写代码之外的一切——Skills 同步、账户切换、用量监控、对话回顾。"

记住三个锚点就够了：

① 定位：连接层，不是又一个 AI 工具。
② 4 大模块：账户与用量 / 文档 / 技能中心 / 工具设置。
③ 哲学：痛点驱动飞轮，不提前规划。
