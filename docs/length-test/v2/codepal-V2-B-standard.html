TL;DR

AI 工具只管写代码，写代码之外的事——同步 Skills、切账户、看花了多少钱、翻历史对话——没人管。CodePal 就是那个"幕后助手"，专门做 Claude Code / Codex / Cursor / Trae 之间跨工具的连接活。一句话定位：AI 写代码，CodePal 管周边。

01故事的起点：一个被忽略的"工具空隙"

不是 AI 不够强，而是 AI 工具之间没人搭桥。

设想一位每天用 AI 写代码的开发者：早上在 Claude Code 里写功能，中午切到 Codex 试另一个模型，晚上在 Cursor 改前端，偶尔还在 Trae 里跑实验。四个工具，四套 Skills 目录，四份配置——改一个 prompt 模板，就要复制粘贴四次；删一个过时的 Skill，要去四个地方各删一遍。

更折磨的是 Codex Plus / Pro 的双窗口限额（5 小时 + 7 天）：额度一触顶，必须 Logout、换邮箱、重登、二次验证，1–2 分钟一次，一天循环 N 次。再要顺手记一下"这个月 AI 帮我花了多少钱、哪个模型最贵、上次和它聊过的那段方案讨论在哪个 Session 里"——没有面板告诉你，只能在终端 history 和 ~/.claude/projects/ 里翻。

REVERSE · 反例
常见误判是"再加一个更聪明的 AI 工具就好了"。但新工具只会让跨工具的事更多——又多一份 Skills、又多一个账户、又多一处 Token 消耗。问题不在 AI 本身，在于没有一个工具愿意管"别人家工具"。Claude Code 不会管 Codex，Codex 不会管 Cursor。

这道空隙，就是 CodePal 站的位置。

02转折：CodePal 做"AI 不愿意做"的四件事

不写一行业务代码，只做工具与工具之间的连接。

CodePal 把日常摩擦拆成四个独立模块，每个模块对应一种"AI 工具不会自己解决"的痛：

① 账户与用量 · 让额度可见、切换无感

多 Codex 账户纯文件交换切换（<200ms 完成，不碰 Codex.app），登录新账号会被自动嗅到（chokidar 监听 ~/.codex/auth.json）、自动提示保存；v1.6.2 起每 24 小时后台续签 token，让"躺着的账户"切回去也不会 401。Claude 会员额度则接 官方 rate_limits，5h / 7d 剩余 + 满载率趋势直接挂状态栏，触顶前一眼看到；Token 花费按日 / 周 / 月、按模型 / 按项目可视化，不用再去翻账单猜账。

② 文档 · 找回那条"昨天聊过的对话"

Claude Code 的所有 Session 按项目分组、全文搜索，找到后一键复制 cd "..." && claude --resume <uuid>，或者直接 osascript 起 Terminal 重启对话——粘到任何终端都能接着聊。每条消息支持 Markdown 渲染（代码块、链接、图片引用都在）。再也不用在 ~/.claude/projects/ 里翻文件夹猜哪个是上周的会话。

③ 技能中心 · 写一次，到处用

从任意工具扫描导入 Skills 到中央仓库，批量推送到所有工具——改一处全局生效。扫描覆盖 ~/.claude/skills/、~/.codex/skills/、~/.cursor/skills/ 加自定义路径，按标签筛选 + 搜索；每 5 分钟自动增量刷新，新 Skill 自动同步出去。MCP 服务也按工具集中开关，不用手动改 JSON。

④ 工具设置 · 起新项目不用再"抄作业"

Claude Code 的四档权限模式（只读规划 / 每次询问 / 自动编辑 / 全自动）一键切换；新项目模板按下生成 CLAUDE.md + .gitignore + 记忆系统 + 可选 git init，配上常驻的 IP 监控和 Anthropic / OpenAI 连通性诊断，挂 VPN 出问题时第一时间定位是网络还是 API 自身。

REVERSE · 反例
容易把 CodePal 误判成"又一个 AI 编程工具"。但它不写代码、不调模型、不抢 IDE 的活——它做的是工具之间的胶水层。如果用一句类比：AI 工具是厨师，CodePal 是后厨那个洗菜、传菜、记账、点存货的帮工。少了帮工，厨师也能炒菜，但每盘菜要慢 2 分钟。

03启示：为什么这是"个人自用驱动"的产品

不规划 roadmap，只解决自己当下的痛。

CodePal 没有功能列表，遵循"用 → 痛 → 解决 → 沉淀"的飞轮——作者云舒每天用 Claude Code / Codex 写代码，碰到一个跨工具的痛，就给 CodePal 加一个模块，做完自己用，再放出来。

这种节奏带来一个特征：每个功能都对应一个真实切肤的痛，没有"为了功能丰富而做"的注水。代价是路线图不可预测——v1.4.5 的"对话回顾"、v1.5.0 的"Codex 多账户"、v1.6.2 的"凭证保活"，每一个都是某个深夜被某件事逼出来的：要找昨天那段对话却找不到、额度又掉了要重登第三次、几天没用的账户切回去 401。

REVERSE · 反例
对 PM 来说，"自用驱动"听起来浪漫，但容易被误读成"不需要做用户研究"。准确的读法应该是：作者本人就是高频用户，反馈回路 < 24 小时。这跟"拍脑袋做功能"完全是两回事——前者是把自己当 design partner，后者是把自己当用户全部。CodePal 走得通，是因为 AI 编程工具用户的痛点高度同构（都在多工具切、都被额度卡、都翻不到对话）。换一个用户群分散的领域，自用驱动就会失效。
PM 视角 · 可复用框架

遇到"跨工具的连接需求"时，问三个问题：
1. 这件事原工具会不会自己解决？（如果会 → 等就行）
2. 不解决的成本是每天发生还是偶尔？（每天 → 才值得做工具）
3. 解决方案能不碰原工具的核心流程吗？（能 → 才不会被原工具升级打破）

CodePal 三问全是 yes，所以飞轮转得起来。

"AI 工具负责写代码，CodePal 负责写代码之外的一切。"

读完手里多了什么

  • 一个口诀：AI 写代码，CodePal 管周边。
  • 一个判断框架：跨工具连接产品的"三问"——原工具会不会管 / 痛是不是天天发生 / 能不能不动原工具核心。
  • 一张定位地图：CodePal 的四块——账户用量、文档回顾、Skills 同步、工具设置——对应"AI 不会自己做"的四类摩擦。

边界声明

CodePal 不是团队级账户管理工具（无云端同步），也不是跨平台方案（Windows / Intel Mac 暂未打包），更不解决 AI 写代码本身的能力问题。它只在"个人开发者 + macOS Apple Silicon + 多 AI 工具并用"这个交集里成立。
